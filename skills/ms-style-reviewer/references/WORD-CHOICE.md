# Word choice

> Source: Microsoft Writing Style Guide — Word choice

Use the same word consistently when you mean the same thing. Choose words wisely for readability and comprehension.

## Use contractions

- Use common contractions (*it's, you're, that's, don't*) to create a friendly, informal tone
- Don't mix contractions and spelled-out equivalents in the same UI — pick one and be consistent
- Never form a contraction from a noun and a verb
  - ✗ *Microsoft's developing a lot of new cloud services.*
- Avoid ambiguous or awkward contractions: *there'd, it'll, they'd*

## Use simple words, concise sentences

- Choose simple verbs without modifiers. Avoid weak or vague verbs (*be, have, make, do*)

  | ✓ Use | ✗ Not |
  |---|---|
  | use | utilize, make use of |
  | remove | extract, take away, eliminate |
  | tell | inform, let know |

- Don't use two or three words when one will do

  | ✓ Use | ✗ Not |
  |---|---|
  | to | in order to, as a means to |
  | also | in addition |
  | connect | establish connectivity |

- Choose words with one clear meaning

  | ✓ Use | ✗ Not |
  |---|---|
  | *Because* you created the table… | *Since* you created the table… |

- Omit unnecessary adverbs unless important to meaning (*quite, very, quickly, easily, effectively*)
- Use one term consistently to represent one concept
- Use words that can be both nouns and verbs carefully (*file, post, mark, screen, record, report*) — use sentence structure and context to eliminate ambiguity

## Don't use common words in new ways

- Use words in their most familiar sense (first dictionary definition)
- Don't create a new word from an existing word
  - ✗ *bucketize* to mean *group*
- Don't apply a new meaning to an ordinary word
  - ✗ *graveyard* to mean *archive*
- Be careful with common words that have industry-specific uses — assume customers know the common definition, not the industry-specific one. Define in context if needed
- Don't use verbs as nouns or nouns as verbs

  | ✓ Use | ✗ Not |
  |---|---|
  | affect performance | impact performance |
  | download the paper | get the download |
  | respond to the request | respond to the ask |

## Use technical terms carefully

- Don't use a technical term when an everyday term will do
  - ✓ *copy* files from a CD — ✗ *rip* files from a CD
- When you must use technical terms, define them in context
- Use technical terms consistently — one term, one concept across all content
- Use industry-specific terms for professional audiences, but verify usage against the A–Z word list and Merriam-Webster first
- Don't create a new term if an existing one serves the purpose
- Research emerging terminology before adopting — verify consistent usage across products and services

## Avoid jargon

- Use a more familiar term when one exists
  - ✓ *symbol* — ✗ *glyph*
- Don't use jargon if the term is familiar only to a small segment of readers
- Don't use jargon for terms that aren't specific to software, networking, or cloud services
- Avoid business, marketing, and journalistic jargon
  - ✗ *leverage* to mean *take advantage of*

### Testing for jargon

- If you think a term is jargon, it probably is
- If it's an acronym or abbreviation, it may be jargon — spell it out for clarity
- If a reviewer questions your use of a term, it may be jargon

## Use US spelling and avoid non-English words

- Use US spelling when spelling varies by locale
  - ✓ *license* — ✗ *licence*
- Avoid non-English words or phrases
  - ✗ *de facto*, *ad hoc*
- Avoid Latin abbreviations for common English phrases

  | ✓ Use | ✗ Not |
  |---|---|
  | for example | e.g. |
  | that is | i.e. |
  | namely | viz. |
  | therefore | ergo |

- **Exception:** *etc.* is OK when space is limited; otherwise prefer *and so on*
